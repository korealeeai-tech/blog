이 블로그의 첫 글은 블로그 환경을 어떻게 만들었는지에 대한 이야기로 시작하려고 한다.

처음에는 AI가 블로그 서비스에 직접 연결해서 글을 만들고, 초안을 저장하고, 필요하면 발행까지 도와주는 흐름을 생각했다. 겉으로 보기에는 자연스럽다. 글은 AI가 쓰고, 블로그는 웹 서비스에 있으니, 둘을 연결하면 바로 자동화가 될 것처럼 보인다.

하지만 실제로는 그 흐름이 생각보다 맞지 않았다. 가장 큰 이유는 AI가 실행되는 환경과 내가 실제로 브라우저를 쓰는 환경이 분리되어 있었기 때문이다.

## 브라우저 기반 자동화가 어색했던 지점

블로그 서비스에 글을 쓰려면 보통 브라우저 로그인 상태가 필요하다. 로그인 세션, 쿠키, 팝업 인증, 계정 전환 같은 것들이 자연스럽게 따라온다.

문제는 AI 작업 도구가 원격 서버에서 실행되고, 사용자는 개인 PC에서 SSH로 접속하는 구조에서는 그 브라우저 상태가 깔끔하게 공유되지 않는다는 점이다. 사용자의 PC 브라우저에는 로그인되어 있어도, 원격 서버의 AI 도구는 그 상태를 그대로 알 수 없다.

이런 구조에서 브라우저 로그인 기반 자동화를 억지로 붙이면 다음 문제가 생긴다.

- 인증 상태를 확인하기 어렵다.
- 로그인과 발행 권한의 경계가 흐려진다.
- 실패했을 때 원인이 브라우저인지, 네트워크인지, 자동화 도구인지 분리하기 어렵다.
- AI가 실제로 무엇을 바꿨는지 diff로 검토하기 어렵다.

즉, 글을 쓰는 자동화보다 로그인 상태를 맞추는 일이 더 큰 일이 될 가능성이 있었다.

## Git 기반 블로그가 더 잘 맞았던 이유

그래서 방향을 바꿨다. 블로그 글을 웹 UI에서 직접 작성하는 대신, Markdown 파일로 관리하고 Git으로 발행하는 구조를 선택했다.

이 방식은 AI 작업과 잘 맞는다. AI는 파일을 만들고 수정하는 데 강하다. Markdown 글을 작성하고, diff를 보여주고, 빌드가 되는지 확인하고, 문제가 없으면 commit/push까지 이어갈 수 있다.

Git 기반 블로그의 장점은 단순하다.

- 글의 변경 내역이 파일 diff로 남는다.
- 발행 전 빌드 검증을 할 수 있다.
- 이전 상태로 되돌리기 쉽다.
- 공개 발행 결과에 들어가면 안 되는 내용이 있는지 검색하기 쉽다.
- 블로그 전용 계정과 SSH 키를 분리해서 운영할 수 있다.

브라우저 자동화는 "지금 로그인되어 있는가"가 중요하지만, Git 기반 자동화는 "어떤 파일이 어떻게 바뀌었는가"가 중요하다. AI와 함께 일할 때는 후자가 훨씬 검토하기 쉽다.

## 지금 블로그의 기본 구조

현재 이 블로그는 Astro 기반 정적 사이트로 만들었다. 글은 Markdown으로 작성하고, GitHub Pages로 배포한다.

운영 흐름은 다음처럼 단순하게 잡았다.

1. AI가 Markdown 글 초안을 작성한다.
2. 공개해도 되는 내용인지 검토한다.
3. 빌드해서 페이지가 생성되는지 확인한다.
4. 변경 파일과 diff를 확인한다.
5. Git으로 commit/push한다.
6. GitHub Pages 배포 결과를 확인한다.

이 흐름에서 중요한 점은 AI가 바로 "웹 화면에 글을 올리는 존재"가 아니라는 것이다. AI는 글과 변경사항을 만들고, 검증 가능한 형태로 남긴다. 발행은 Git 이력과 빌드 결과를 통해 확인한다.

## 첫 번째 기준

이 블로그는 AI 설정, 작업 흐름, 모델 사용 경험, 시행착오를 기록하는 공간으로 쓰려고 한다.

첫 번째 기준은 단순하다. 자동화가 편하더라도 검토할 수 없는 자동화는 쓰지 않는다. AI가 만든 결과는 파일, diff, 빌드 결과, 배포 결과로 확인할 수 있어야 한다.

그래서 첫 글도 블로그 도구 자체에 대한 이야기로 시작했다. 앞으로는 AI 작업 환경을 어떻게 조정했는지, 어떤 설정이 실제 작업 품질에 영향을 줬는지, 어떤 실패를 통해 검증 기준을 바꿨는지를 하나씩 정리해볼 생각이다.
