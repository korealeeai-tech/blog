“이거 확인해줘”라는 요청을 받았다고 하자. 문장은 짧고 자연스럽지만, 해야 할 일은 하나로 정해지지 않는다.

현재 파일이 있는지만 보고할 수 있다. 사용자의 원인 가설이 맞는지 반례까지 조사할 수도 있다. 문제가 확인되면 바로 수정하고 배포까지 이어갈 수도 있다. 세 행동은 모두 `확인`과 양립하지만 권한, 비용, 완료 조건이 다르다.

이 글은 AI가 실제 내부에서 어떻게 의도를 계산하는지 설명하지 않는다. 그 메커니즘은 이 합성 예제로 확인할 수 없다. 대신 **짧은 요청 하나가 여러 작업 계약과 동시에 양립하는 정보 부족 문제**를 다룬다. 무엇을 기억하게 할지보다, 어떤 신호를 어떤 순서로 적용하고 언제 질문해야 하는지가 핵심이다.

## 같은 문장에 세 계약을 넣어본다

`확인해줘`를 세 방식으로 구체화하면 차이가 선명해진다.

| 계약 | 해야 할 일 | 완료 증거 | 하지 말아야 할 일 |
|---|---|---|---|
| 관찰 | 현재 상태와 검색 범위를 확인 | 경로, 명령 출력, 발견 여부 | 요청 없는 수정 |
| 가설 검증 | 주장 분해, 찬반 근거, 반례 확인 | confirmed/refuted/uncertain과 gap | 근거 부족을 동의로 덮기 |
| 수정 실행 | 원인 확인 뒤 변경·검증 | diff, test/build, 남은 위험 | 범위 밖 외부 write |

사용자가 어떤 계약을 원하는지 말하지 않았는데 AI가 하나를 고르면, 결과가 좋아도 의도와 어긋날 수 있다. 관찰만 원했는데 파일을 고치면 과잉 실행이고, 수정까지 원했는데 상태만 알려주면 미완료다.

문제는 단어 하나의 뜻을 사전에서 더 정확히 찾는 것으로 해결되지 않는다. 같은 표현이 상황에 따라 실제로 다른 계약을 가리키기 때문이다.

## 현재 요청과 지속 기준은 역할이 다르다

지속 memory나 instruction은 반복 설명을 줄이는 데 도움이 될 수 있다. OpenAI Codex memories 문서도 이전 작업의 useful context를 다음 작업으로 가져올 수 있다고 설명한다.

> [OpenAI Codex memories 문서](https://developers.openai.com/codex/memories): "carry useful context"

지속 기준은 default를 제공한다. 예를 들어 “단순 상태 확인 요청은 read-only로 답한다”, “명시적 실행 요청이면 검증까지 끝낸다”는 기준이 있으면 모호한 요청의 기본 행동이 안정된다.

하지만 default는 authority가 아니다. 현재 사용자가 “수정하지 말고 원인만 알려줘”라고 명시했다면 과거의 자동 수정 기준을 적용하면 안 된다. 반대로 사용자가 “문제면 바로 고쳐줘”라고 했는데 과거의 보수적 read-only 선호만 따르는 것도 틀린다.

안전한 우선순위는 다음과 같다.

```text
1. 현재 요청의 명시적 범위와 금지사항
2. 현재 작업·저장소에 적용되는 구체 규칙
3. 반복해서 확인된 지속 default
4. 여전히 남는 고위험 모호성에 대한 확인 질문
```

기억이 많을수록 3번은 풍부해질 수 있다. 그렇다고 1번과 2번을 건너뛸 수는 없다.

## 모든 모호성에 질문하면 또 다른 문제가 생긴다

조금이라도 해석 가능성이 여러 개라고 매번 질문하면 안전해 보인다. 하지만 read-only로 쉽게 확인할 수 있는 일까지 멈추면 사용자는 반복해서 허가를 줘야 한다.

예를 들어 “파일이 어디 있는지 찾아줘”는 검색 범위가 저장소 안으로 한정돼 있고 변경이 없다면 먼저 `rg --files`로 확인할 수 있다. 후보가 둘이면 그때 결과를 보여주고 범위를 좁히면 된다.

질문 없이 진행해도 되는 조건은 대체로 이렇다.

- read-only이고 현재 scope 안에서 끝난다.
- 잘못된 가정을 해도 즉시 되돌릴 수 있다.
- 로컬 근거를 읽으면 해석이 하나로 좁혀진다.
- 결과를 제시한 뒤 사용자가 다음 결정을 할 수 있다.

반대로 다음 경우에는 한 번 확인하는 편이 낫다.

- 삭제, 외부 공개, 비용, 운영 변경처럼 되돌리기 어렵다.
- 두 해석이 서로 다른 사람·시스템·권한에 영향을 준다.
- 어느 쪽을 선택하느냐에 따라 산출물의 목적이 달라진다.
- 로컬 근거를 읽어도 사용자의 가치 판단 없이는 정할 수 없다.

질문 횟수를 줄이는 것이 목표가 아니다. **질문이 필요한 모호성과 조사로 해소할 수 있는 모호성을 구분하는 것**이 목표다.

## 지속 기준이 오히려 틀리게 만드는 반례

과거에 사용자가 짧은 답변을 여러 번 원했다고 하자. 이를 “사용자는 항상 짧은 답변을 원한다”로 저장하면, 상세한 법적 검토나 설계 비교 요청에서도 요약만 내놓을 수 있다.

좋은 지속 기준은 범위와 예외를 가진다.

```text
나쁜 기준:
사용자는 짧은 답변을 원한다.

더 나은 기준:
단순 상태 확인은 짧게 답한다.
분석·설계·검증 요청은 근거와 한계를 분리해 필요한 깊이로 답한다.
현재 요청이 다른 형식을 명시하면 현재 요청을 따른다.
```

첫 문장은 기억하기 쉽지만 현재 요청을 덮는다. 두 번째는 적용 조건이 있어 retrieval 뒤에 판단할 수 있다.

이 때문에 persistent memory를 사용자 성격 요약으로 만들면 위험하다. 다음 작업의 행동을 바꾸는 scoped criterion으로 바꾸고, 현재 요청과 충돌하는지 매번 확인해야 한다.

## 현재 근거도 memory가 대신할 수 없다

작업 범위를 잘 골랐다고 사실까지 맞는 것은 아니다.

예를 들어 “현재 branch를 확인해줘”라는 의도를 정확히 이해했어도, 과거 session 기록을 읽고 branch를 답하면 stale할 수 있다. 이 요청의 근거는 방금 실행한 `git branch --show-current`여야 한다.

따라서 두 질문을 분리해야 한다.

```text
어떻게 일할 것인가? → 현재 요청 + scoped rule + 지속 default
무엇이 사실인가?     → 현재 파일 + 공식 문서 + 방금 실행한 결과
```

memory는 첫 질문의 출발점을 도울 수 있다. 두 번째 질문의 source proof를 대신하지 않는다.

## 반복 설명을 줄이는 최소 구조

지속 기준을 운영하려면 거대한 사용자 요약부터 만들 필요가 없다. 반복되는 기준을 다음처럼 좁게 적을 수 있다.

```text
상황: 단순 상태 확인
default: read-only로 관찰하고 범위를 보고한다.
예외: 사용자가 수정·실행을 명시하면 해당 범위까지 진행한다.
질문 조건: 외부 write나 파괴적 행동이 두 해석 중 하나에만 포함될 때 묻는다.
fresh proof: 현재 파일과 명령 결과를 다시 확인한다.
```

이 구조는 사용자를 묘사하지 않는다. 특정 상황에서 AI가 어떤 default를 쓰고 언제 멈출지 설명한다.

지속 기준을 더 체계적으로 분류하는 방법은 [SecondBrain: AI를 위한 사용자 의도 라이브러리](/blog/blog/secondbrain-user-intent-library/)에서 다룬다. 그 글은 선호·검증·위험·작업 맥락·응답 전략의 구성에 집중하고, 이 글은 짧은 요청 하나가 여러 행동 계약과 양립할 때의 처리 순서에 집중한다.

## 결론

짧은 요청으로 작업 범위를 정하기 어려운 이유는 AI가 무조건 사용자를 기억하지 못해서가 아니다. `확인해줘` 같은 문장이 관찰, 가설 검증, 수정 실행이라는 서로 다른 계약과 동시에 양립하기 때문이다.

지속 context는 default를 제공해 반복 설명을 줄일 수 있다. 하지만 현재 명시 요청보다 우선하면 안 되고, 현재 사실의 근거를 대신해서도 안 된다. 조사로 범위를 좁힐 수 있으면 먼저 확인하고, 되돌리기 어려운 복수 해석이 남을 때 질문한다.

이 글이 보여준 것은 언어와 작업 계약의 모호성이다. 특정 모델이 실제로 어떤 내부 과정으로 의도를 추론하는지는 검증하지 않았다. 그 경계를 지켜야 “AI가 왜 그런가”라는 설명이 확인하지 않은 모델 심리 묘사로 흐르지 않는다.
