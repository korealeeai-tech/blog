AI 에이전트 지시문을 정리할 때 “몇 줄까지가 적당한가”부터 물으면 답이 잘 나오지 않는다. 삭제하면 안전 기준이 사라질 수 있고, 남겨 두면 현재 작업과 관계없는 절차까지 매번 읽게 된다. 줄 수보다 먼저 정해야 할 것은 각 지시의 자리다.

이 글의 질문은 하나다. **어떤 지시를 상시 규칙, 필요할 때 읽는 절차, 자동 검증 gate 중 어디에 둘 것인가.** 특정 길이가 성능을 높이거나 낮춘다고 주장하지 않고, 합성 instruction inventory를 세 기준으로 끝까지 분류해 본다.

여기서 확인되는 것은 필요할 때 지침을 불러오고 자동 처리를 연결할 수 있다는 기능 범위, 그리고 한 제품 문서가 긴 memory 파일에 제시하는 주의점까지다. 세 계층 분류가 실제 작업 품질을 얼마나 바꾸는지는 이 출처들만으로 알 수 없다.

## 배치 판단에 쓰는 세 질문

지시마다 다음 세 질문을 같은 순서로 묻는다.

1. **작업 관련성:** 거의 모든 작업에서 필요한가, 특정 trigger가 있을 때만 필요한가.
2. **충돌 가능성:** 다른 작업이나 더 가까운 규칙에서는 반대 행동이 필요할 수 있는가.
3. **자동 검증 가능성:** 사람이 의미를 해석하지 않아도 pass/fail을 재현할 수 있는가.

작업 관련성이 넓고 충돌 가능성이 낮으며 판단이 필요한 내용은 상시 규칙에 가깝다. 특정 trigger에서만 관련되거나 환경에 따라 순서가 달라지는 내용은 필요 시 절차로 보낸다. 입력과 결과로 실패를 재현할 수 있는 조건은 자동 gate의 후보가 된다.

이것은 점수표가 아니다. 예를 들어 보안 경계는 자동 검색도 가능하지만, 검색에 잡히지 않는 의미 노출은 사람이 판단해야 한다. 자동화 가능성 하나만 보고 원래 판단 기준을 모두 삭제하면 안 된다.

## 합성 inventory 일곱 개를 실제로 나눠 본다

가상의 `sample-blog` 저장소에 다음 지시가 한 파일로 쌓였다고 하자. 실제 프로젝트에서 가져온 목록이 아니라 배치 과정을 보여주기 위한 합성 사례다.

| 지시 | 작업 관련성 | 충돌 가능성 | 자동 판정 | 주된 자리 |
|---|---|---|---|---|
| 응답은 한국어로 하고 기술 식별자는 원문으로 둔다 | 거의 모든 작업 | 낮음 | 의미 판단 필요 | 상시 규칙 |
| 요청 밖 파일과 기존 변경을 보존한다 | 거의 모든 수정 | 낮음 | diff로 일부만 확인 | 상시 규칙 |
| 배포 전에 preview, 링크, 원격 상태를 순서대로 확인한다 | 배포 작업만 | 환경별 절차 차이 있음 | 일부 가능 | 필요 시 배포 절차 |
| 외부 문헌 캡처는 주소창을 제외하고 확인일과 한계를 적는다 | 글·문서 작업만 | 다른 산출물에는 무관 | 일부 가능 | 필요 시 글쓰기 절차 |
| 모든 게시물에 `heroImage`가 있어야 한다 | 게시물 검증 시 | 낮음 | schema로 가능 | 자동 gate |
| 공개 결과물에 개인·credential·식별 가능한 맥락을 넣지 않는다 | 공개 작업 전반 | 낮음 | pattern으로 일부만 가능 | 상시 판단 규칙 + 보조 scan gate |
| 지난달 한 번 쓴 임시 배포 우회 명령을 항상 실행한다 | 현재 관련성 확인 안 됨 | 현재 절차와 충돌 가능 | 실행은 가능하나 정당성 없음 | 삭제 또는 이력 보관 |

첫 두 항목은 작업 종류가 달라져도 행동 경계를 바꾸므로 상시 규칙에 남긴다. 세 번째와 네 번째는 trigger가 명확하다. 상시 파일에는 “배포 작업이면 배포 절차를 읽는다” 같은 진입점만 두고, 세부 순서는 해당 절차가 맡는다.

다섯 번째는 자연어로 여러 번 강조할 이유가 적다. schema가 누락을 정확히 실패시킬 수 있기 때문이다. 여섯 번째와 마지막 항목은 한 칸에 넣는 판정만으로 책임과 수명을 설명할 수 없다. 두 항목은 아래에서 근거와 실패 모드까지 따로 판정한다.

## 세 가지 정리 전략은 각각 쓸 곳이 있다

inventory를 정리하는 선택지는 세 가지다.

| 전략 | 유리한 조건 | 감수할 실패 |
|---|---|---|
| 낡은 지시를 모두 삭제 | 저장소가 작고 현재 코드·test가 유일한 기준이며 복구 가능한 이력이 있음 | 판단과 승인 경계까지 함께 사라질 수 있음 |
| 모든 내용을 상시 로드 | 규칙이 몇 개뿐이고 거의 모든 작업에 적용되며 충돌도 드묾 | 상황별 절차와 오래된 예외가 현재 작업을 방해할 수 있음 |
| 상시 규칙·필요 시 절차·자동 gate로 분리 | 작업 종류가 여러 개이고 trigger와 기계 판정 조건을 구분할 수 있음 | routing 링크 누락, gate 오탐, 계층 간 중복을 따로 관리해야 함 |

세 계층 분리가 항상 정답은 아니다. 규칙이 다섯 줄뿐인 작은 저장소라면 한 파일이 더 투명할 수 있다. 반대로 기존 지시가 거의 전부 폐기된 도구의 임시 우회책이라면 과감한 삭제가 맞다. 세 계층은 **유효한 지시는 남겨야 하지만 모든 작업에 항상 필요하지는 않은 상황**에서 가장 의미가 있다.

## 두 항목은 일대일 분류로 끝나지 않는다

inventory를 실제로 정리하면 “상시냐, 절차냐, gate냐” 중 하나만 고르는 것으로 끝나지 않는 항목이 나온다. 같은 위험에 두 방어가 필요할 수도 있고, 현재 활성 지시에서는 삭제하되 다시 살릴 증거를 정의해야 할 수도 있다.

### 공개 민감정보 규칙: 주된 판단과 보조 방어를 나눈다

`공개 결과물에 개인·credential·식별 가능한 맥락을 넣지 않는다`는 항목의 주된 자리는 공개 작업에 적용되는 상시 판단 규칙이다. 본문뿐 아니라 파일명, 이미지, caption, 생성 결과를 함께 보고, 여러 단서를 조합했을 때 대상을 유추할 수 있는지도 판단해야 하기 때문이다. 이 책임은 정규식으로 완전히 표현하기 어렵다.

scan gate는 같은 문장을 복사하는 두 번째 지시가 아니다. 알려진 credential 형식, 로컬 경로, 내부 host 후보처럼 기계적으로 찾을 수 있는 신호를 반복 검사하고, 발견 위치와 실패 상태를 반환하는 보조 방어다. scan 0건은 “알려진 pattern을 찾지 못했다”는 뜻이지 의미상 공개 안전을 증명하지 않는다.

| 자리 | 맡는 책임 | 이 자리만으로 알 수 없는 것 |
|---|---|---|
| 주된 상시 판단 규칙 | 어떤 표면과 의미 조합을 공개 위험으로 볼지 결정하고 애매하면 보류 | 모든 raw pattern이 실제로 검색됐는지 |
| 보조 scan gate | 정해진 pattern을 source·asset·생성 결과에서 같은 방식으로 찾고 누락 시 실패 | 문맥을 조합한 재식별 가능성, 이미지의 암시 |

따라서 이중 배치는 같은 규칙을 두 번 강조하는 것이 아니라 서로 다른 실패 모드를 맡기는 판정이다. 판단 규칙을 scan으로 대체하면 의미 누출을 놓치고, scan 없이 판단에만 맡기면 반복 가능한 raw leak을 사람 눈에 의존하게 된다.

### 오래된 우회 명령: 활성 지시에서는 빼고 부활 증거를 정한다

`지난달 한 번 쓴 임시 배포 우회 명령을 항상 실행한다`는 항목은 자동 실행이 가능하다는 이유만으로 gate가 되지 않는다. 현재 환경에서 같은 문제가 재현된다는 근거가 없고, 정상 경로와 충돌할 수 있으므로 활성 지시에서는 삭제한다. 원래 증상, 적용 당시 버전, 제거 이유는 날짜가 붙은 이력에 남겨 나중에 원인을 비교할 수 있게 한다.

이 지시를 다시 활성화하려면 최소한 현재 지원 환경에서 원래 증상이 재현되어야 한다. 여기에 현재 primary source가 workaround의 필요성을 뒷받침하거나, focused 비교에서 명령 없이 실패하고 명령을 적용했을 때만 통과한다는 증거 중 하나가 더 필요하다. 다시 넣을 때도 적용 trigger와 만료·재확인 조건을 함께 정한다. 비슷한 오류 문구가 한 번 보였다는 사실만으로는 부활시키지 않는다.

두 판정은 placement가 문서 구조를 한 번 정하는 일이 아니라는 점을 보여준다. inventory의 각 항목마다 현재 관련성, 충돌 가능성, 자동 판정 범위, 근거 수명을 다시 봐야 한다. 그 결과 한 항목은 주된 자리와 보조 방어를 함께 갖고, 다른 항목은 활성 세 계층 어디에도 남지 않을 수 있다.

관련 글과의 경계도 이 지점에 있다. [운영 규칙의 scope와 경로 계층](/blog/blog/agents-md-ai-operating-rules/)은 이미 세 계층의 역할과 global·repo·하위 경로 scope, 변경 후 readback을 다룬다. 이 글은 그 구조를 다시 설명하는 대신 **합성 inventory를 항목별로 판정하고, 애매한 이중 배치와 stale 지시의 삭제·부활 증거까지 결정한다.** [규칙과 검증 gate의 차이](/blog/blog/ai-rules-need-validation-gates/)가 일반적인 강제 장치를 설명한다면, 여기서는 민감정보 항목 하나에서 판단 규칙과 scan gate의 책임이 왜 다른지만 적용한다.

## 분류가 실패하는 조건

trigger가 부정확하면 필요한 절차가 로드되지 않는다. gate가 너무 좁으면 잘못된 결과를 통과시키고, 너무 넓으면 오탐 때문에 우회 욕구를 만든다. 이중 배치한 항목의 책임을 구분하지 않고 같은 문장을 조금씩 다르게 복사하면 어느 쪽이 최신인지 다시 모호해진다.

합성 inventory의 판정도 고정 답안은 아니다. “외부 문헌 캡처”가 매일 수행하는 핵심 업무인 환경에서는 상시 진입점을 더 구체적으로 둘 수 있고, `heroImage`가 선택 필드인 사이트에서는 해당 gate가 틀린 요구사항이 된다. 배치는 실제 task 빈도와 현재 계약을 다시 확인해야 한다.

## 결론과 한계

지시문 정리의 핵심은 세 칸을 만드는 데서 끝나지 않고 각 항목의 책임과 수명을 판정하는 일이다. 대부분은 주된 자리 하나를 갖지만, 공개 민감정보 항목처럼 상시 판단과 scan gate가 서로 다른 실패를 막을 수도 있다. 현재 근거가 없는 우회 명령은 활성 지시에서 삭제하고, 재현과 현재 근거가 생길 때만 scope와 만료 조건을 붙여 되살린다.

이 글은 일곱 개 합성 지시를 구조적으로 분류했을 뿐, 같은 모델과 task를 두 조건에서 실행한 비교 실험은 아니다. 세 계층 분리가 token, 비용, 정확도, 준수율을 얼마나 바꾸는지는 측정하지 않았다. 실제 적용에서는 inventory를 먼저 만들고, 누락된 절차와 gate 오탐을 관찰하며 배치를 다시 조정해야 한다.
