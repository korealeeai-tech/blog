AI 에이전트를 오래 쓰다 보면 지시문 파일이 점점 길어진다. 처음에는 좋은 의도다. 같은 실수를 반복하지 않게 하고 싶고, 프로젝트 규칙을 알려주고 싶고, 공개하면 안 되는 정보도 막고 싶다.

문제는 어느 순간부터 지시문이 안전장치가 아니라 소음이 된다는 점이다. AI가 매번 읽어야 할 문맥은 늘어나는데, 정작 중요한 규칙은 긴 문장 사이에 묻힌다. 사람도 긴 회의록에서 핵심을 놓치듯이, AI도 길고 모호한 지시문을 항상 같은 품질로 따르지는 않는다.

그래서 나는 지시문에도 다이어트가 필요하다고 본다. 줄인다는 말은 규칙을 약하게 만든다는 뜻이 아니다. 항상 읽어야 할 것, 필요할 때만 읽을 것, 문서가 아니라 도구로 강제할 것을 나누자는 뜻이다.

이 글은 2026-07-01 기준으로 OpenAI Codex와 Claude Code 공식 문서를 확인하고, 공개 가능한 운영 원칙으로 정리한 내용이다. 특정 도구의 내부 구현을 추정하지 않고, 공식 문서에서 확인 가능한 범위와 개인적인 판단을 분리해서 쓴다.

## 지시문이 길어지는 이유

지시문이 길어지는 이유는 대체로 비슷하다.

AI가 실수한다. 그러면 사람은 "다음부터는 이것도 조심해"라고 적는다. 또 다른 실수가 생기면 한 줄을 더 넣는다. 어느 날 보면 `AGENTS.md`나 `CLAUDE.md`가 운영 매뉴얼, 회고록, 체크리스트, 예외 규칙, 개인 선호, 프로젝트 설명을 모두 떠안고 있다.

이 방식은 초반에는 효과가 있다. 하지만 계속 쌓이면 문제가 생긴다.

첫째, 서로 다른 층위의 규칙이 섞인다. "한국어로 답하라" 같은 상시 규칙과 "블로그 이미지를 추가할 때는 원본 크기로 확인하라" 같은 상황별 절차가 같은 무게로 놓인다.

둘째, 추상적인 문장이 늘어난다. "품질 좋게 작성한다", "보안에 유의한다", "사용자 의도를 잘 반영한다" 같은 문장은 맞는 말이지만 행동으로 옮기기 어렵다.

셋째, 실제로 강제해야 할 항목까지 지시문에만 남는다. 예를 들어 secret scan, build, schema validation은 문서에만 쓰기보다 명령으로 확인하는 편이 낫다.

## 공식 문서에서 확인한 경계

Codex 문서에서는 `AGENTS.md`가 전역과 프로젝트 지침을 계층적으로 읽는 방식으로 설명된다. skills는 반복 워크플로우를 묶어 필요할 때 불러오는 장치이고, hooks는 특정 이벤트에 맞춰 스크립트나 처리를 연결하는 확장 지점이다. memories는 도움이 되는 문맥을 가져오는 보조 계층으로 볼 수 있다.

Claude Code 문서도 비슷한 경계를 알려준다. `CLAUDE.md`와 auto memory는 세션 시작 시 문맥으로 로드되지만, 강제 설정은 아니다. 문서에서는 구체적이고 간결한 지시가 더 일관되게 따르기 쉽고, 긴 지시문은 context를 더 쓰며 준수율을 떨어뜨릴 수 있다고 설명한다.

내가 여기서 얻은 기준은 이렇다. 지시문 파일은 모든 것을 넣는 창고가 아니라, 작업 시작 시 반드시 떠올려야 하는 기준만 담는 곳이어야 한다.

## 항상 읽어야 할 규칙만 남긴다

상시 지시문에는 짧고 반복적인 기준만 남기는 것이 좋다.

예를 들면 이런 것들이다.

- 응답 언어와 톤
- 추측 금지와 확인 방식
- public repo 안전 경계
- 변경 전 기존 패턴을 읽는 원칙
- 완료 전 검증과 보고 방식
- 절대 하지 말아야 할 destructive 행동

이 기준은 작업 종류가 바뀌어도 거의 항상 필요하다. 그래서 매번 문맥에 들어가도 비용 대비 효과가 있다.

반대로 다음 항목은 상시 지시문에 길게 넣지 않는 편이 낫다.

- 특정 배포 절차
- 특정 문서 양식 전체
- 긴 예시 코드
- 오래된 회고
- 한 번만 필요했던 예외
- 특정 화면이나 기능의 상세 구현 설명

이런 내용은 별도 protocol, skill, README, 체크리스트로 빼는 편이 더 낫다.

## 절차는 필요할 때 불러오게 한다

AI가 매번 모든 절차를 읽을 필요는 없다. 블로그 글을 쓸 때는 블로그 workflow가 필요하고, 코드 리뷰를 할 때는 리뷰 기준이 필요하고, 배포를 할 때는 배포 체크리스트가 필요하다.

이런 문서는 상시 지시문에서 짧게 연결만 해두면 된다.

```md
블로그 작업은 docs/protocols/blog-workflow.md를 따른다.
보안 검토 작업은 docs/protocols/security-review.md를 먼저 읽는다.
```

이렇게 하면 상시 문맥은 가벼워지고, 필요한 작업에서만 세부 절차가 들어온다. 중요한 것은 파일을 나누는 것 자체가 아니라, 언제 어떤 문서를 읽어야 하는지 명확히 하는 것이다.

AI에게 "필요하면 알아서 찾아봐"라고만 하면 놓칠 수 있다. 대신 "이 작업 유형이면 이 문서를 읽어라"처럼 연결 규칙을 남겨야 한다.

## 강제할 수 있는 것은 문서 밖으로 빼낸다

지시문은 AI에게 방향을 알려준다. 하지만 지시문만으로 모든 것을 막을 수는 없다.

예를 들어 "대표 이미지를 반드시 넣어라"는 규칙은 content schema로 확인할 수 있다. "빌드가 깨지면 발행하지 말라"는 규칙은 `npm run build`로 확인할 수 있다. "민감정보를 넣지 말라"는 규칙은 사람이 보는 diff 검토와 함께 pattern scan을 돌릴 수 있다.

이런 항목은 문서에만 남기지 말고 검증 명령으로 옮기는 것이 좋다.

내가 선호하는 분리는 다음과 같다.

- 판단 기준: 지시문
- 반복 절차: protocol 또는 skill
- 형식 요구사항: schema
- 품질 확인: test, lint, build
- 위험 탐지: secret scan, 금지어 검색, 이미지 원본 확인

문서가 줄어드는 대신 검증이 늘어나야 한다. 그래야 지시문 다이어트가 통제 약화가 아니라 통제 구조 개선이 된다.

## 나쁜 지시문과 좋은 지시문

나쁜 지시문은 대체로 크고 부드럽다.

```md
항상 고품질로 작성하고 보안에 주의한다.
```

틀린 말은 아니지만 AI가 무엇을 해야 하는지 분명하지 않다.

좋은 지시문은 작고 확인 가능하다.

```md
public repo에 올리기 전 source, assets, dist를 대상으로 회사명, 개인식별정보, 내부 경로, token 패턴을 검색한다.
```

이 문장은 길어 보여도 행동이 분명하다. 어떤 범위를 봐야 하는지, 무엇을 찾는지, 언제 실행하는지 알 수 있다.

지시문을 줄일 때 기준은 문장 수가 아니다. "이 문장이 실제 행동을 바꾸는가"다. 행동을 바꾸지 못하는 문장은 삭제하거나, 더 구체적인 체크로 바꿔야 한다.

## 오래된 규칙을 정리한다

지시문은 한 번 만들고 끝나는 문서가 아니다. 도구가 바뀌고, 작업 방식이 바뀌고, 이전에 중요했던 위험이 더 이상 중요하지 않을 수 있다.

그래서 가끔은 지시문을 읽고 이런 질문을 해야 한다.

- 지금도 항상 필요한가?
- 특정 작업에만 필요한가?
- 이미 테스트나 schema로 강제하고 있지 않은가?
- 서로 충돌하는 규칙은 없는가?
- 너무 추상적이어서 검증할 수 없는가?
- 공개 repo에 적기에는 과한 내부 맥락을 담고 있지 않은가?

이 질문에 걸리는 문장은 줄이거나 옮긴다. 특히 개인적인 작업 기록이나 특정 업무 맥락은 공개 가능한 일반 원칙으로 바꾸지 못하면 넣지 않는 편이 맞다.

## 내가 쓰는 결론

AI 에이전트를 잘 쓰려면 지시문을 많이 쓰는 것보다 정확히 나눠야 한다.

항상 읽어야 하는 규칙은 짧게 둔다. 긴 절차는 필요할 때 불러오게 한다. 강제 가능한 요구사항은 schema, test, build, scan으로 옮긴다. 그리고 지시문은 실제 작업에서 반복되는 실수를 줄일 때만 추가한다.

AI에게 모든 것을 말로 부탁하는 방식은 오래 버티기 어렵다. 좋은 운영은 지시문, 절차 문서, 자동 검증, 사람 검토가 각자 자기 역할을 갖는 상태에 가깝다.

지시문 다이어트의 목적은 AI를 덜 통제하는 것이 아니다. 중요한 규칙이 더 잘 보이게 만드는 것이다.
