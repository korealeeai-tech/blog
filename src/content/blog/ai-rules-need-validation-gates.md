AI에게 일을 맡기다 보면 한 번쯤 이런 말을 하게 된다.

"규칙에 적어놨는데 왜 안 지켰지?"

처음에는 지시문을 더 자세히 쓰면 해결될 것처럼 보인다. 빠진 기준을 한 줄 더 넣고, 예외를 더 적고, 다음부터는 반드시 지키라고 강조한다. 그런데 시간이 지나면 비슷한 일이 다시 생긴다. AI가 규칙을 아예 못 본 것은 아닌데, 실제 작업 순간에는 그 규칙이 행동으로 이어지지 않는 경우가 있다.

나는 이 문제를 지시문 품질만의 문제로 보지 않는다. 더 정확히는 "읽히는 규칙"과 "통과 조건"을 같은 것으로 착각했을 때 생기는 문제에 가깝다.

이 글은 특정 도구의 내부 동작을 추정하려는 글이 아니다. OpenAI Codex와 Claude Code 공식 문서에서 확인 가능한 범위와, AI 작업을 운영하면서 얻은 개인적인 기준을 분리해서 쓴다.

## 규칙을 읽는다는 것은 강제한다는 뜻이 아니다

먼저 공식 문서에서 확인한 경계를 짧게 보면 좋다.

> [OpenAI Codex AGENTS.md 문서](https://developers.openai.com/codex/guides/agents-md): "Codex reads `AGENTS.md` files before doing any work."

이 문장은 `AGENTS.md`가 작업 전 문맥으로 들어간다는 근거다. 나는 이 지점이 중요하다고 본다. 지시문 파일은 AI가 작업을 시작할 때 참고할 기준을 제공한다. 답변 언어, 수정 방식, 검증 기준, 공개하면 안 되는 정보 같은 기본값을 정할 수 있다.

<figure>
	<img src="/blog/blog-images/official-docs/openai-codex-agents-md-reads.png" alt="OpenAI Codex AGENTS.md 공식문서에서 Codex가 작업 전 AGENTS.md 파일을 읽는다고 설명한 영역 캡처" />
	<figcaption>OpenAI Codex AGENTS.md 문서 캡처, 캡처일 2026-07-01, 링크 재확인 2026-07-06. 작업 전 지침 로드 근거를 보여주지만, 지침이 모든 행동을 강제로 보장한다는 뜻은 아니다.</figcaption>
</figure>

하지만 "읽는다"와 "그대로 실행된다"는 다르다. 지시문은 모델의 판단에 영향을 주는 문맥이다. 그 문맥이 길거나, 모호하거나, 현재 요청과 충돌하거나, 검증 명령으로 이어지지 않으면 실제 결과에서는 빠질 수 있다.

Claude Code memory 문서도 비슷한 경계를 더 직접적으로 말한다.

> [Claude Code memory 문서](https://code.claude.com/docs/en/memory): "context, not enforced configuration"

나는 이 문장을 AI 운영에서 꽤 중요한 기준으로 읽는다. memory나 instruction file은 유용한 출발점이지만, 그것만으로 위험한 행동을 막거나 누락을 자동으로 잡아내지는 않는다.

	<figcaption>Claude Code memory 문서 캡처, 캡처일 2026-07-01, 링크 재확인 2026-07-06. memory와 지시문이 참고 문맥이라는 경계를 보여주지만, 모든 AI 도구가 같은 방식으로 동작한다는 증거는 아니다.</figcaption>
</figure>

## 지시문이 실패하는 흔한 이유

지시문을 잘 써도 실패하는 이유는 여러 가지가 있다.

첫 번째는 지시가 너무 추상적일 때다.

```md
품질 좋게 작성한다.
보안에 주의한다.
출처를 잘 확인한다.
```

틀린 말은 아니지만, 작업자가 무엇을 해야 하는지 분명하지 않다. AI에게도 마찬가지다. "잘"이라는 말은 행동으로 바꾸기 어렵다. 좋은 규칙은 더 작고 확인 가능해야 한다.

```md
외부 문서를 인용하면 본문 인용문 바로 아래에 공식문서 캡처 figure를 둔다.
캡션에는 확인일과 캡처가 증명하지 않는 것을 같이 적는다.
검증 명령이 실패하면 commit/push하지 않는다.
```

두 번째는 규칙이 실제 완료 조건과 연결되지 않을 때다. 예를 들어 "대표 이미지를 반드시 넣는다"라고 적어도, 빌드나 검증 스크립트가 대표 이미지 누락을 실패로 만들지 않으면 누락될 수 있다.

세 번째는 현재 요청의 속도가 규칙보다 앞설 때다. 사용자가 "바로 해줘"라고 하면 AI는 실행을 우선하려고 할 수 있다. 이때 지시문이 길고 검증 경로가 자동화되어 있지 않으면, 중요한 항목이 뒤로 밀린다.

네 번째는 완료 보고가 결과물 중심일 때다. 파일이 생기면 완료라고 말하기 쉽다. 하지만 공개 글이라면 파일 생성만으로는 부족하다. 이미지 경로, 외부 출처 캡처, 빌드 결과, 민감정보 검색, live page까지 봐야 한다.

## 강제해야 하는 것은 실패하게 만든다

그래서 나는 규칙을 두 종류로 나누는 편이 낫다고 본다.

하나는 판단 기준이다. 이것은 문서에 적는다. 글의 톤, 공개 범위, 어떤 내용을 과장하지 말아야 하는지, 확인한 것과 추정을 어떻게 분리할지 같은 기준이다.

다른 하나는 통과 조건이다. 이것은 가능하면 검증 명령으로 만든다. 대표 이미지가 없으면 실패하고, 외부 문헌 인용 옆 캡처가 없으면 실패하고, 빌드가 깨지면 실패하고, 민감한 문자열이 잡히면 실패해야 한다.

OpenAI Codex hooks 문서는 이런 자동화의 한 축을 설명한다.

> [OpenAI Codex hooks 문서](https://developers.openai.com/codex/hooks): "Hooks are an extensibility framework for Codex."

이 문장을 내가 적용하는 방식은 단순하다. AI에게 "잊지 말라"고 반복하기보다, 반복 확인이 필요한 부분은 hook, script, test, build, schema 같은 실패 가능한 장치로 옮긴다.

물론 모든 규칙을 자동화할 수는 없다. 글이 자연스러운지, 독자가 오해할 만한 표현이 있는지, 출처 해석이 과한지, 이미지가 실제 증거처럼 보이지 않는지는 사람이 봐야 한다. 자동 검증은 사람이 볼 일을 없애는 장치가 아니라, 반복적으로 빠지는 기본 조건을 먼저 걸러내는 장치에 가깝다.

## 내가 쓰는 분리 기준

나는 AI 작업 규칙을 만들 때 다음처럼 나눈다.

- 문서에 둘 것: 작업 철학, 공개 범위, 판단 기준, 위험 경계
- 스크립트로 둘 것: 누락 여부, 파일 존재, 링크 형식, 빌드, 정적 검색
- 사람이 볼 것: 글의 설득력, 과장 여부, 캡처의 의미, 공개 안전의 애매한 경계

예를 들어 블로그 글쓰기라면 문서에는 "공식문서를 근거로 쓰면 원문 캡처와 한계를 함께 적는다"라고 둔다. 그리고 검증 스크립트에는 "외부 출처 인용 blockquote 뒤에 `official-docs` 이미지가 없으면 실패" 같은 조건을 둔다.

이렇게 하면 AI가 규칙을 놓쳤을 때도 결과가 조용히 통과하지 않는다. 누락은 실패 메시지로 드러난다. 사람이 봐야 할 문제와 자동으로 막을 문제도 분리된다.

## 좋은 규칙은 짧고, 좋은 게이트는 차갑다

지시문은 사람이 읽어도 이해되어야 한다. 너무 길고 감정적인 규칙은 나중에 관리하기 어렵다.

```md
실수하지 않게 잘 검증한 뒤 안전하게 처리한다.
```

이런 문장은 마음은 이해되지만 실제로는 약하다. 무엇을 검증해야 하는지, 실패하면 무엇을 멈춰야 하는지, 어떤 증거를 남겨야 하는지 알 수 없다.

반대로 좋은 게이트는 차갑다.

```text
heroImage가 없으면 실패
캡처 파일이 없으면 실패
확인 기준 링크가 본문 근거와 연결되지 않으면 실패
build가 실패하면 push 금지
```

이런 문장은 친절하지 않아도 된다. 대신 빠뜨렸을 때 바로 멈춰야 한다.

## 오해하기 쉬운 지점

검증 게이트를 만든다고 AI가 모든 규칙을 자동으로 지키는 것은 아니다. 게이트가 보는 것은 게이트에 적힌 조건뿐이다. 그래서 검증 스크립트가 통과했다고 해서 글의 논리가 좋거나, 이미지 품질이 충분하거나, 공개해도 안전하다는 결론까지 자동으로 나오지는 않는다.

또 하나 조심할 점은 게이트가 너무 넓어지는 경우다. 모든 것을 정규식으로 잡으려 하면 오탐이 늘고, 실제 작업은 느려진다. 자동 검증은 반복적으로 빠지고, 실패 비용이 크고, 기계적으로 확인 가능한 항목부터 붙이는 편이 낫다.

내 기준에서는 다음 항목이 좋은 자동화 후보였다.

- 필요한 파일이 존재하는가
- frontmatter가 schema를 만족하는가
- 외부 인용 옆에 근거 캡처가 있는가
- 캡션에 확인일과 한계가 있는가
- 빌드가 통과하는가
- 공개하면 안 되는 문자열 후보가 남아 있는가

반대로 다음 항목은 사람이 봐야 한다.

- 글이 실제 경험을 바탕으로 자연스럽게 읽히는가
- 출처에서 끌어낸 결론이 과하지 않은가
- 캡처가 실제 증거보다 더 많은 것을 증명하는 것처럼 보이지 않는가
- 공개 가능한 일반론으로 충분히 바뀌었는가

## 내가 얻은 결론

AI에게 규칙을 적는 일은 필요하다. 하지만 규칙을 적는 것만으로 충분하다고 보면 위험하다.

지시문은 AI가 작업을 시작할 때 기준을 떠올리게 한다. 검증 게이트는 그 기준 중 일부를 통과 조건으로 바꾼다. 사람 검토는 자동화가 보지 못하는 의미와 맥락을 확인한다.

이 셋은 서로 대체 관계가 아니다. 지시문만 있으면 놓칠 수 있고, 검증만 있으면 의미를 보지 못하고, 사람 검토만 있으면 반복 작업에서 피로가 쌓인다.

그래서 내가 남긴 기준은 이렇다.

AI에게 "규칙을 지켜라"라고만 말하지 않는다. 중요한 규칙은 통과 조건으로 만든다. 그리고 검증이 통과했다는 사실과 사람이 판단해야 하는 부분을 끝까지 분리한다.

그때 비로소 규칙은 문서 안의 좋은 말이 아니라, 실제 작업을 멈추거나 통과시키는 기준이 된다.
