AI에게 코드를 맡길 때 가장 빠른 방법은 바로 수정하라고 하는 것이다. 파일을 열고, 바꾸고, 테스트를 돌리면 된다. 작은 작업에서는 이 방식이 꽤 잘 먹힌다.

하지만 조금만 복잡해지면 문제가 생긴다. AI는 모르는 부분을 자연스럽게 채우려고 한다. 프로젝트의 규칙, 기존 패턴, 공식 문서의 경계, 검증 방법이 비어 있으면 그 빈칸을 추측으로 메울 수 있다. 결과가 그럴듯해 보여도 실제 repo와 맞지 않는 경우가 생긴다.

그래서 나는 코드 수정 전에는 문서부터 읽히는 편이다. 이것을 거창한 절차로 만들려는 것이 아니다. AI가 추측해야 하는 범위를 줄이기 위한 최소한의 준비다.

이 글은 2026-07-01 기준 OpenAI Codex 공식 문서와 내가 실제로 쓰는 공개 가능한 작업 방식을 바탕으로 정리했다. 특정 조직이나 프로젝트의 내부 규칙이 아니라, AI 코딩 작업을 안전하게 시작하기 위한 일반 원칙에 초점을 맞춘다.

## 먼저 확인한 것

OpenAI Codex 문서에서는 `AGENTS.md`를 작업 전 읽는 지침 파일로 설명한다. 또 skills 문서는 필요한 지침을 필요한 때 불러오는 progressive disclosure 방식을 설명한다.

> [OpenAI Codex AGENTS.md 문서](https://developers.openai.com/codex/guides/agents-md): "Codex reads `AGENTS.md` files before doing any work."

나는 이 문장을 "코드 수정 전에 repo의 기본 규칙을 먼저 읽혀야 한다"는 근거로 본다. 다만 이 문장만으로 `AGENTS.md`가 모든 실수를 막는다고 해석하면 안 된다. 문서는 시작점이고, 실제 검증은 따로 필요하다.

## 코드부터 보면 빠르지만 좁게 본다

AI가 바로 코드를 보면 눈앞의 문제에는 빨리 반응한다. 함수 이름을 바꾸고, 조건문을 추가하고, 오류가 난 줄을 고치는 일은 빠르게 처리할 수 있다.

문제는 그 코드가 왜 그렇게 생겼는지 모를 때다. 같은 이름의 helper가 있는지, 비슷한 화면이나 모듈에서는 어떻게 처리했는지, 이 repo에서 금지하는 패턴이 있는지 모르면 AI는 일반적인 방식으로 답을 만들 수 있다.

일반적인 방식이 항상 틀린 것은 아니다. 하지만 프로젝트에는 프로젝트의 이유가 있다. 오래된 호환성, 빌드 제약, 배포 방식, 테스트 관례, 공개 안전 기준 같은 것들은 코드 한 조각만 봐서는 잘 보이지 않는다.

그래서 코드부터 보는 방식은 빠르지만 좁다. 문서를 먼저 보는 방식은 조금 느리지만, 무엇을 추측하면 안 되는지 알려준다.

## 문서는 AI의 시야를 제한한다

문서를 읽힌다는 말은 AI에게 더 많은 정보를 주는 것처럼 들린다. 하지만 내가 기대하는 효과는 반대에 가깝다. AI가 멋대로 상상할 수 있는 범위를 줄이는 것이다.

예를 들어 작업 전에 다음을 확인하게 한다.

- 이 repo에서 반드시 지켜야 하는 응답과 작업 규칙
- 수정 대상 주변의 기존 구현 패턴
- 공식 문서가 말하는 기능의 실제 범위
- 이번 작업에서 하지 않을 범위
- 완료 전에 확인할 검증 명령

이렇게 하면 AI가 "보통은 이렇게 한다"가 아니라 "이 repo에서는 이렇게 해야 한다"에 가깝게 움직인다.

물론 문서가 많다고 좋은 것은 아니다. 오래된 문서, 서로 충돌하는 문서, 너무 긴 지시문은 오히려 판단을 흐릴 수 있다. 그래서 문서도 우선순위가 필요하다. 현재 요청에 직접 관련된 문서부터 읽고, 관계가 약한 문서는 보류해야 한다.

## 공식 문서와 프로젝트 문서는 역할이 다르다

공식 문서는 도구나 라이브러리가 무엇을 지원하는지 알려준다. 하지만 우리 작업에서 어떤 선택을 해야 하는지는 공식 문서만으로 정해지지 않는다.

프로젝트 문서는 반대로 그 repo에서 어떤 방식을 쓰기로 했는지 알려준다. 예를 들어 commit 기준, 이미지 사용 기준, 테스트 명령, 공개 금지사항 같은 것은 프로젝트 문서가 더 중요할 수 있다.

둘 중 하나만 보면 빈틈이 생긴다. 공식 문서만 보면 프로젝트의 관례를 놓칠 수 있고, 프로젝트 문서만 보면 도구의 실제 동작을 잘못 이해할 수 있다.

그래서 나는 보통 이 순서로 본다.

1. 공식 문서로 기능의 경계를 확인한다.
2. 프로젝트 문서로 작업 규칙을 확인한다.
3. 주변 코드로 실제 관례를 확인한다.
4. 이번 작업의 성공 기준과 제외 기준을 짧게 고정한다.
5. 그 다음에 코드를 수정한다.

이 순서가 매번 필요한 것은 아니다. 하지만 변경 범위가 넓거나 공개 결과물이 생기는 작업에서는 이 순서를 생략하지 않는 편이 낫다.

## 작업 계약을 먼저 짧게 만든다

문서를 읽힌 뒤 바로 코드를 고치게 하지 않는다. 중간에 작업 계약을 짧게 만든다.

작업 계약은 복잡할 필요가 없다. 이번 작업에서 무엇을 바꾸고, 무엇은 바꾸지 않고, 어떤 검증을 통과해야 하는지를 몇 줄로 정리하면 된다.

예를 들면 이런 식이다.

```text
목표: 새 블로그 글 1편을 추가한다.
제외: 내부 업무 사례, 회사/제품/개인 정보는 쓰지 않는다.
근거: 공식 문서 인용 시 원문 캡처와 확인일을 포함한다.
검증: build, diff check, 민감정보 검색, live URL 확인.
```

이 계약이 있으면 AI가 작업 중간에 방향을 잃었을 때 다시 돌아올 기준이 생긴다. 사람도 결과를 볼 때 무엇을 기준으로 판단해야 하는지 분명해진다.

## 문서를 읽어도 검증은 필요하다

Doc-first는 검증을 대체하지 않는다. 문서를 먼저 읽어도 AI가 잘못 이해할 수 있고, 문서가 최신 코드와 맞지 않을 수도 있다.

그래서 문서를 읽힌 뒤에는 실제 증거를 다시 봐야 한다. 코드 변경이면 테스트나 빌드가 필요하고, 글 발행이면 생성된 HTML과 live page 확인이 필요하다. 공식 문서를 인용했다면 원문 링크와 확인일도 남겨야 한다.

문서를 먼저 읽히는 목적은 "이제 안심해도 된다"가 아니다. 검증할 대상을 더 정확히 고르는 것이다.

## 내가 남긴 기준

AI에게 코드를 맡기기 전에 문서부터 읽히는 이유는 절차를 늘리기 위해서가 아니다. AI가 추측해야 하는 빈칸을 줄이기 위해서다.

작업 전에는 공식 문서, 프로젝트 규칙, 주변 패턴, 성공 기준을 먼저 본다. 그 다음에 수정한다. 마지막에는 실제 빌드와 diff, 필요한 경우 live 결과로 확인한다.

작은 작업에서는 이 흐름을 짧게 압축할 수 있다. 하지만 되돌리기 어렵거나 공개되는 작업이라면 코드부터 고치는 속도보다, 무엇을 근거로 고칠지 정하는 시간이 더 중요하다.

내가 얻은 결론은 단순하다. AI에게 문서를 먼저 읽히는 것은 느린 습관이 아니라, 나중에 틀린 수정을 되돌리는 시간을 줄이는 습관이다.
