AI 작업 기억을 만들기 시작하면 가장 먼저 하고 싶은 일이 있다. 많이 저장하는 것이다. 대화도 저장하고, 작업 기록도 저장하고, 자주 쓰는 표현도 저장하면 AI가 나를 더 잘 이해할 것처럼 느껴진다.

하지만 나는 이제 그 방식이 위험하다고 본다. 기억이 많아질수록 좋아지는 것이 아니라, 무엇을 기억으로 남길지 정하지 않으면 오히려 AI가 더 쉽게 헷갈린다. 오래된 맥락을 현재 사실처럼 쓰거나, 공개하면 안 되는 내용을 요약이라는 이름으로 다시 꺼낼 수도 있다.

그래서 SecondBrain 같은 개인 AI 작업 기억을 만들 때는 저장소를 먼저 생각하지 않는다. 저장 경계를 먼저 정한다. 무엇을 남기고, 무엇은 조심해서 남기고, 무엇은 애초에 저장하지 않을지 나누는 것이다.

이 글은 공개 가능한 개념으로만 설명한다. 실제 개인 기록, 내부 경로, 원문 로그, 업무 화면, 계정 정보, 비공개 작업 맥락은 다루지 않는다. 2026-07-01 기준 OpenAI Codex와 Claude Code 공식 문서에서 memory 관련 설명을 확인했고, 그 위에 개인적인 운영 기준을 덧붙인다.

## 먼저 확인한 것

OpenAI Codex 문서는 memories를 이전 작업에서 유용한 context를 다음 작업으로 가져오는 장치로 설명한다. 이 설명은 memory가 현재 판단을 대신한다는 뜻이 아니다. 어디까지나 다음 작업에 도움이 되는 문맥을 가져오는 보조 계층으로 읽는 편이 안전하다.

> [OpenAI Codex memories 문서](https://developers.openai.com/codex/memories): "Memories let Codex carry useful context"

나는 이 문장을 memory에 무엇이든 넣어도 된다는 뜻으로 보지 않는다. 오히려 다음 작업에 유용한 context만 남겨야 한다는 기준으로 본다.

	<img src="/blog/blog-images/official-docs/openai-codex-memories-context.png" alt="OpenAI Codex memories 공식문서에서 Codex가 유용한 context를 다음 작업으로 가져올 수 있다고 설명한 영역 캡처" />
	<figcaption>OpenAI Codex memories 문서 캡처, 확인일 2026-07-01. memory가 문맥을 이어주는 장치라는 근거이며, 모든 기록을 저장해야 한다는 뜻은 아니다.</figcaption>
</figure>

Claude Code 문서도 memory를 강제 설정이 아니라 context로 구분한다. 이 경계는 중요하다. memory는 AI가 참고할 수 있는 문맥이지, 현재 작업의 사실 검증이나 행동 제한을 자동으로 보장하지 않는다.

> [Claude Code memory 문서](https://code.claude.com/docs/en/memory): "Claude treats them as context, not enforced configuration."

그래서 memory에 저장할 내용은 "AI가 반드시 믿어야 하는 사실"보다 "현재 작업에서 다시 확인해야 할 기준"에 가까워야 한다.

## 저장해야 할 것: 반복되는 행동 기준

AI 작업 기억에 가장 먼저 남길 것은 원문이 아니라 행동 기준이다.

예를 들어 "사용자는 꼼꼼하다"라는 기억은 별로 도움이 되지 않는다. 꼼꼼하다는 말은 넓고, 작업마다 의미가 달라진다. 대신 이렇게 바꾸면 쓸모가 생긴다.

```text
검증한 것과 검증하지 못한 것을 최종 보고에 분리한다.
```

이 문장은 다음 작업에서 바로 쓸 수 있다. AI는 결과를 보고할 때 어떤 형식으로 확인해야 하는지 알 수 있고, 사람도 그 기준이 지켜졌는지 판단할 수 있다.

좋은 작업 기억은 보통 이런 형태다.

- 답변 방식: 짧게 답할 때와 근거를 길게 설명할 때를 구분한다.
- 검증 기준: 완료 전에 어떤 증거를 확인해야 하는지 남긴다.
- 위험 기준: 반복해서 나온 실수와 중단 조건을 남긴다.
- 공개 기준: 글, 이미지, 파일명, build output까지 공개 표면으로 본다.
- 작업 방식: 큰 작업은 성공 기준과 제외 기준을 먼저 맞춘다.

이런 기억은 사용자를 분석하는 문장이 아니라, 다음 작업에서 AI가 취할 행동을 정하는 문장이다.

## 조심해서 저장할 것: 시간이 지나면 바뀌는 기준

어떤 기억은 저장할 수 있지만 조심해야 한다. 시간이 지나면 바뀌거나, 상황에 따라 달라질 수 있는 것들이다.

예를 들어 도구 설정, 선호하는 작업 흐름, 자주 쓰는 검증 명령, 글쓰기 규칙은 유용하다. 하지만 이 정보는 바뀔 수 있다. 그래서 확인일, 적용 범위, 예외를 함께 남기는 편이 안전하다.

나쁜 예시는 이런 식이다.

```text
항상 이 방식으로 발행한다.
```

좋은 예시는 조금 더 좁다.

```text
공개 블로그 글은 작성 시점 기준으로 build, diff check, live URL 확인을 수행한다.
단, 배포 방식이 바뀌면 확인 절차를 다시 본다.
```

이렇게 쓰면 memory가 낡았을 때도 위험이 줄어든다. AI가 기억을 절대 규칙으로 받아들이기보다, 현재 작업에서 다시 확인할 기준으로 다루게 된다.

## 저장하지 않을 것: 원문과 식별 가능한 맥락

가장 중요한 부분은 저장하지 않을 것을 정하는 일이다.

개인 AI 작업 기억을 만들 때 원문 대화 전체를 계속 쌓고 싶어질 수 있다. 하지만 원문은 정보가 너무 많다. 그 안에는 개인적인 표현, 계정 정보, 내부 맥락, 특정 작업을 떠올리게 하는 단서가 섞일 수 있다.

나는 이런 것은 작업 기억에 넣지 않는 편이 안전하다고 본다.

- 원문 대화 전체
- 개인 이름, 계정, 이메일, 연락처
- 내부 경로, private repository, issue 번호
- 회사, 제품, 고객, 조직을 유추할 수 있는 작업 맥락
- 로그, token, key, cookie, session 정보
- 실제 화면 캡처나 원본 파일 내용
- 오래된 추정이나 확인되지 않은 결론

여기서 중요한 것은 "잘 익명화해서 저장하자"가 아니다. 익명화된 구조 자체가 맥락을 드러낼 수 있다. 필요 없는 원문은 애초에 저장하지 않는 편이 낫다.

## 원문 대신 판단 기준으로 바꾼다

원문을 저장하지 않는다고 해서 경험을 버리는 것은 아니다. 중요한 것은 원문에서 다음 작업에 쓸 수 있는 판단 기준만 뽑아내는 것이다.

예를 들어 어떤 작업에서 AI가 확인하지 않은 내용을 완료라고 말해 문제가 생겼다고 하자. 이때 저장할 것은 사건의 원문이 아니다. 저장할 것은 재발 방지 기준이다.

```text
완료 보고 전에는 실행한 검증과 실행하지 못한 검증을 분리해서 적는다.
```

이렇게 바꾸면 민감한 맥락 없이도 다음 작업에 도움이 된다. 기억은 과거 사건의 사본이 아니라, 다음 작업의 안전장치가 된다.

SecondBrain을 안전하게 만들려면 이 변환이 중요하다. 기록을 그대로 쌓는 것이 아니라, 공개 가능한 수준의 작업 기준으로 압축하는 것이다.

## 좋은 기억은 현재 요청을 덮지 않는다

memory가 많아지면 또 다른 문제가 생긴다. AI가 현재 요청보다 기억을 더 믿는 것이다.

예를 들어 과거에는 긴 설명을 선호했지만, 지금 사용자가 "짧게 말해줘"라고 했다면 현재 요청이 우선이다. 과거에 특정 검증 방식을 썼더라도, 지금 도구나 배포 방식이 바뀌었다면 다시 확인해야 한다.

그래서 좋은 기억은 단정형보다 확인형에 가깝다.

```text
사용자는 항상 긴 설명을 원한다.
```

보다

```text
분석 요청에서는 근거와 한계를 분리해 설명하는 편이 좋다. 단, 사용자가 짧게 요청하면 짧게 답한다.
```

가 낫다.

memory는 현재 요청을 이기는 규칙이 아니다. 현재 요청을 더 잘 이해하기 위한 힌트다.

## 내가 쓰는 저장 기준

나는 AI 작업 기억을 남길 때 보통 네 가지 질문을 한다.

1. 다음 작업에서 실제 행동을 바꿀 수 있는가?
2. 원문 없이도 의미가 유지되는가?
3. 개인, 조직, 계정, 비공개 맥락을 유추할 단서가 없는가?
4. 시간이 지나도 그대로 믿어도 되는가, 아니면 확인일과 한계가 필요한가?

이 질문을 통과하지 못하면 memory로 남기지 않는다. 필요하면 더 일반적인 기준으로 바꾸거나, 아예 버린다.

좋은 SecondBrain은 많은 것을 기억하는 구조가 아니다. 필요한 것을 안전하게 기억하고, 위험한 것은 저장하지 않는 구조다.

## 내가 남긴 기준

AI 작업 기억을 만들 때 가장 중요한 것은 저장 기술이 아니라 저장 경계다.

저장할 것은 반복되는 행동 기준, 검증 기준, 위험 기준이다. 조심해서 저장할 것은 시간이 지나면 바뀌는 도구 설정과 작업 흐름이다. 저장하지 않을 것은 원문 대화, 개인 정보, 비공개 맥락, 식별 가능한 단서다.

memory는 AI가 사용자를 더 많이 저장하기 위한 장치가 아니다. 다음 작업에서 덜 추측하고, 더 안전하게 확인하기 위한 기준표에 가깝다.

내가 얻은 결론은 단순하다. AI가 나를 기억하게 만들기 전에, 무엇을 기억하지 말아야 하는지부터 정해야 한다.
