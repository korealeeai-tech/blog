AI가 만든 결과를 받을 때 설명부터 읽으면 “왜 이렇게 했는가”에 먼저 설득되기 쉽다. 하지만 인수 여부를 결정하는 것은 설명의 매끄러움이 아니라 실제 산출물이다.

이 글의 질문은 **AI 설명을 읽기 전에 실제 산출물에서 어떤 순서로 위험을 찾는가**다. 일반적인 항목만 나열하지 않고, 의도적으로 다섯 가지 결함을 심은 [공개용 합성 변경 보고서](/blog/blog-examples/review-output-sample.md)를 같은 순서로 끝까지 리뷰한다.

fixture의 `[L01]`부터 `[L08]`은 이 글에서 줄을 정확히 가리키기 위한 label이다. 나머지 대괄호 표기는 실제 값을 대신하는 sentinel이다. 검토 결과부터 말하면 이 산출물은 인수할 수 없다.

## 1. 요청과 결과를 먼저 맞춘다 — `L01`과 `L02`

fixture의 `L01`은 요청을 “도움말 문구 한 문장만 고친다”로 고정한다. 그런데 `L02`는 도움말뿐 아니라 화면 색상 설정도 바꿨다고 적는다.

첫 단계의 판정은 **요청 불일치**다. 색상 변경이 보기 좋거나 build에 문제가 없는지는 아직 중요하지 않다. 사용자가 허용한 결과의 범위를 벗어났기 때문이다.

이 단계에서 할 일은 두 가지 중 하나다.

- 색상 설정 변경을 되돌리고 문구 수정만 남긴다.
- 색상 변경이 꼭 필요하다면 이유와 영향을 설명하고 범위 확장을 먼저 확인한다.

요청 일치를 먼저 보는 이유는 이후 review의 기준점을 만들기 위해서다. 잘못된 방향의 결과를 세밀하게 검토하면, 품질을 높인 뒤에도 인수할 수 없는 산출물이 남는다.

## 2. 요청 밖의 변경을 찾는다 — `L02`와 `L03`

`L02`의 색상 설정은 요청을 넘어선 변경이다. `L03`은 더 분명하다. 예제 작성 순서를 소개하는 단락은 도움말 문구 수정과 관계가 없다고 fixture 자체가 밝힌다.

두 번째 단계의 판정은 **무관 변경 포함**이다. 요청 불일치와 비슷해 보이지만 보는 단위가 다르다. 첫 단계는 최종 결과가 목표와 맞는지 보고, 두 번째는 산출물 안의 각 변경이 그 목표에 필요한지 본다.

`L03`이 문장 자체로 훌륭해도 제거 대상이다. 관련 없는 변경은 검토해야 할 표면을 늘리고, 나중에 문제가 생겼을 때 어떤 변경이 원인이었는지 분리하기 어렵게 한다. 이 fixture는 `L02`의 범위 밖 설정과 `L03`의 무관 문단을 제외한 뒤 다시 diff를 만들어야 한다.

## 3. 검증 결과와 완료 주장을 연결한다 — `L04`와 `L05`

`L04`는 evidence register에 `npm run build` 성공 하나를 남긴다. build 증거의 인식론적 범위는 [AI가 내 결론을 반복하지 않게 검증하는 법](/blog/blog/ai-claim-verification-with-ai/)에서 다룬다. 여기서는 그 범위를 다시 계산하지 않고, fixture의 완료 보고를 인수할 수 있는지만 본다.

`L05`는 등록된 증거 한 건을 내용 정확성, 링크, 공개 안전, live 반영까지 넓힌다. 그런데 `L07`은 live page를 열지 않았다고 직접 밝힌다. 나머지 세 항목도 이 fixture에는 확인 기록이 없다.

따라서 `L05`의 완료 판정은 인수할 수 없다. fixture의 보고는 다음처럼 고쳐야 한다.

```text
확인: build 성공(L04)
미확인: 내용 정확성, 링크, 공개 안전
미수행: live readback(L07)
artifact acceptance: hold
```

## 4. 공개 전 제거 대상을 찾는다 — `L06`

`L06`에는 `[SYNTHETIC_PRIVATE_PATH_REMOVED]`가 있다. 이것은 실제 경로나 비밀값이 아니라, 공개 전에 제거해야 할 자리를 나타내기 위해 만든 명시적 sentinel이다.

네 번째 단계의 판정은 **제거 대상 잔존**이다. sentinel이라서 유출 사고는 아니지만, 최종 public artifact에 남아도 된다는 뜻은 아니다. 실제 작업에서는 이런 자리에 식별 가능한 값이 들어 있었을 가능성을 생각하고 source와 generated output을 함께 확인해야 한다.

이 fixture에서는 `L06`을 삭제하거나 공개 가능한 일반 설명으로 바꾼 뒤, 같은 sentinel이 다른 파일과 생성 결과에 남지 않았는지 검색해야 한다. build가 통과했다는 `L04`는 이 검사를 대신하지 않는다.

## 5. 미확인 위험을 인수 조건과 연결한다 — `L07`과 `L08`

`L07`은 배포된 page를 열지 않았고, `L08`은 generated output과 이미지 metadata를 확인하지 않았다고 밝힌다. 미확인 상태를 숨기지 않았다는 점은 좋다. 그러나 위험을 적었다는 사실과 인수 가능한 상태는 다르다.

다섯 번째 단계의 판정은 **미확인 위험 때문에 public 인수 보류**다. 다음 행동은 위험마다 다르다.

- `L07`: 배포 뒤 실제 page를 열어 제목과 수정 문구를 readback한다.
- `L08`의 generated output: source에 없던 문자열이나 깨진 링크가 생기지 않았는지 확인한다.
- `L08`의 이미지 metadata: 공개하면 안 되는 단서가 없는지 별도로 확인한다.

이 단계에서 “남은 위험이 있습니다”라고 말하고 끝내면 review가 행동으로 이어지지 않는다. 위험이 인수를 막는지, 후속 작업으로 넘겨도 되는지, 누가 확인할지를 정해야 한다. 이 사례에서는 public 발행과 직접 연결되므로 세 항목 모두 보류 조건이다.

## 다섯 단계가 만든 최종 판정

fixture를 실제 줄에 연결하면 다음과 같이 요약된다.

| 순서 | 잡은 줄 | 발견한 문제 | 바로 할 일 |
|---|---|---|---|
| 요청 일치 | `L01–L02` | 요청하지 않은 설정 변경 | 되돌리거나 범위 확인 |
| 변경 범위 | `L02–L03` | 범위 밖 설정과 무관 문단 | 관련 없는 변경 제거 |
| 검증 증거 | `L04–L05` | build보다 넓은 완료 단정 | 주장을 약화하고 별도 검증 |
| 공개 전 제거 | `L06` | 제거 대상 sentinel 잔존 | source·output에서 제거 확인 |
| 남은 위험 | `L07–L08` | live·output·metadata 미확인 | 확인 전 public 인수 보류 |

첫 두 단계만으로도 재작업이 필요하다는 사실을 알 수 있다. 그래서 세부 문체나 미관 review를 먼저 하지 않는다. 범위를 바로잡은 다음 build와 위험 검증을 다시 연결하는 편이 review 비용을 줄인다.

## 설명 우선과 단일 검사 우선은 왜 놓치는가

AI의 완료 설명을 먼저 읽는 방법은 작업 의도를 이해하기에는 유리하다. 하지만 `L05`처럼 근거보다 넓은 주장이 들어 있으면 그 framing을 따라가기 쉽다. 실제 `L01–L03`을 먼저 읽어야 요청과 범위의 차이가 보인다.

반대로 보안 scan 하나를 가장 먼저 실행하는 방법은 알려진 문자열 후보를 빠르게 찾을 수 있다. 그래도 `L01–L03`의 요청 불일치와 `L04–L05`의 논리적 간격은 잡지 못한다. build만 먼저 보는 방법도 같은 이유로 부족하다.

따라서 이 순서는 모든 검사를 한 도구로 해결하려는 방식이 아니다. 산출물의 인수 결정을 싼 질문에서 비싼 질문으로 좁혀 가는 방식이다. 방향과 범위가 틀리면 먼저 돌려보내고, 그 둘이 맞을 때 검증·공개 안전·잔여 위험을 더 깊게 본다.

## 결론: 설명보다 산출물을 이 순서로 본다

AI가 만든 결과를 리뷰할 때는 요청 일치, 변경 범위, 검증 증거, 공개 전 제거 대상, 남은 위험 순서로 실제 artifact를 본다. 합성 fixture는 각 단계에서 `L01–L08`의 구체적인 결함을 드러냈고, 최종 판정은 인수 보류였다.

이 순서의 목적은 AI의 설명을 무시하는 것이 아니다. 설명이 사실인지 대조할 기준을 먼저 확보하는 것이다. artifact를 확인한 뒤 설명을 읽으면 의도, 결과, 증거 사이의 차이를 더 정확히 물을 수 있다.

review에서 수정 대상을 찾았다는 사실만으로 파일을 바꿀 권한이 생기지는 않는다. 다음 질문은 어떤 행동까지 허용됐는가이며, [AI 사실확인 요청의 경계는 단어가 아니라 권한으로 정한다](/blog/blog/ai-factcheck-trigger-boundary/)에서 목적·허용 행동·실패 비용으로 그 경계를 정한다.

## 확인 범위와 한계

- fixture는 review 순서를 재현하기 위해 만든 합성 Markdown이며 실제 작업 결과나 실제 비밀값을 포함하지 않는다.
- 다섯 단계는 일반적인 artifact 인수 순서다. 코드의 정확성, 접근성, 저작권, 규정 준수 같은 surface별 review는 별도로 필요하다.
- 보안 사고, 법률 판단, 운영 배포처럼 실패 비용이 큰 결과에서는 이 순서가 전문 review, 승인, incident 절차를 대체하지 않는다.
- 명백한 위험 신호가 먼저 보이면 순서를 지키기 위해 대응을 늦추지 않는다. 노출이나 장애 가능성이 있으면 containment와 전문 절차가 우선이다.
