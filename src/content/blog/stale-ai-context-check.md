AI memory는 편하다. 매번 같은 선호, 규칙, 작업 맥락을 설명하지 않아도 되기 때문이다.

하지만 memory가 있다고 해서 항상 더 정확해지는 것은 아니다. 오래된 memory는 때때로 위험하다. 어제 맞았던 상태가 오늘은 틀릴 수 있고, 이전 작업의 결론이 새 요청에는 맞지 않을 수 있다. 이 문제를 나는 stale context 문제라고 본다.

stale context는 AI가 틀린 사실을 자신 있게 말하게 만든다. 더 까다로운 점은 그 말이 그럴듯하다는 것이다. 실제로 한 번은 맞았던 정보였기 때문이다.

이 글은 AI memory를 사용할 때 오래된 맥락을 바로 믿지 않기 위해 쓰는 공개 가능한 점검 기준을 정리한 것이다. 특정 memory 제품이나 내부 시스템 설명이 아니라, AI와 작업할 때의 운영 습관에 가깝다.

## memory는 사실이 아니라 후보 근거다

AI memory를 쓸 때 가장 먼저 정해야 할 기준은 이것이다.

memory는 사실 그 자체가 아니라 후보 근거다.

예를 들어 memory에 "이 작업은 이런 방식으로 검증한다"는 기준이 있을 수 있다. 그 기준은 유용하다. 하지만 오늘 작업에도 그대로 적용되는지는 따로 봐야 한다. 작업 규칙이 바뀌었을 수 있고, 이번 요청은 예외일 수 있고, 사용자가 이미 다른 지시를 했을 수도 있다.

memory를 결론으로 쓰면 위험해진다. memory를 출발점으로 쓰면 도움이 된다.

차이는 작지만 중요하다.

```text
나쁜 사용: 예전 memory에 이렇게 되어 있으니 이번에도 그렇다.
좋은 사용: 예전 memory에는 이렇게 되어 있다. 이번 요청과 최신 상태에 맞는지 확인하자.
```

## 마지막 상태가 현재 상태는 아니다

AI가 자주 틀리는 지점은 "마지막으로 본 상태"를 "현재 상태"처럼 말하는 것이다.

예전에 branch가 clean이었다고 해서 지금도 clean인 것은 아니다. 지난번 글 목록이 최신이었다고 해서 오늘도 최신인 것은 아니다. 어제 작업이 완료됐다고 해서 관련 후속 작업이 모두 끝난 것도 아니다.

그래서 상태를 말해야 하는 작업에서는 최신 증거가 필요하다.

- 현재 파일이 실제로 있는가
- 최근 commit이나 변경이 무엇인가
- 작업 상태 문서가 최신인가
- 빌드나 테스트가 지금도 통과하는가
- 공개 페이지나 배포 결과가 지금도 열리는가

이런 질문은 memory가 대신해줄 수 없다. 직접 확인해야 한다.

## 현재 요청과 충돌하는지 본다

오래된 memory는 현재 요청과 충돌할 수 있다.

예를 들어 memory에는 "수정 전에는 반드시 사용자 확인을 받는다"는 기준이 있을 수 있다. 그런데 사용자가 이번에는 "바로 적용해줘"라고 명시했을 수 있다. 반대로 memory에는 "작업 완료 시 commit한다"는 기준이 있지만, 이번 요청에서는 "커밋하지 말고 초안만 만들어줘"라고 했을 수 있다.

이럴 때 AI가 예전 기준만 따르면 사용자의 현재 의도를 놓친다.

그래서 memory를 읽은 뒤에는 반드시 현재 요청을 다시 봐야 한다.

```text
현재 요청이 예전 기준을 강화하는가?
현재 요청이 예전 기준을 좁히는가?
현재 요청이 예전 기준과 충돌하는가?
충돌한다면 어느 쪽이 더 최근이고 더 명시적인가?
```

대체로 현재 사용자의 명시 지시가 더 강하다. 다만 삭제, 외부 발행, 비용 발생, 운영 변경처럼 위험한 작업은 현재 요청이 짧더라도 경계를 다시 확인하는 편이 안전하다.

## 여러 근거를 대조한다

stale context를 줄이려면 하나의 memory만 보지 않는 것이 좋다.

작업 종류에 따라 대조할 근거가 달라진다.

- 현재 사용자 요청
- 최신 작업 기록
- 실제 파일과 diff
- 빌드나 테스트 결과
- 공개 페이지나 API 응답
- 관련 문서의 최신 날짜

이 근거들이 같은 방향을 가리키면 판단이 강해진다. 서로 충돌하면 멈춰야 한다. 충돌을 무시하고 memory 하나만 믿으면 오래된 결론을 현재 사실처럼 말하게 된다.

특히 "다음 작업이 뭐지?", "이어서 해줘", "최근 상태 기준으로 정리해줘" 같은 요청은 stale context 위험이 높다. 이런 요청에서는 마지막 요약 하나만 보면 안 된다. 최근 기록과 실제 상태를 함께 봐야 한다.

## 오래된 memory를 버리는 기준

모든 memory를 계속 들고 있을 필요는 없다. 어떤 기억은 시간이 지나면 버리거나 낮은 우선순위로 내려야 한다.

나는 다음 조건에 걸리면 stale 후보로 본다.

- 생성 시점이 오래됐고 이후 상태 변화가 많다.
- 특정 일회성 작업에만 맞는 기준이었다.
- 현재 문서나 코드와 충돌한다.
- 사용자의 최근 지시와 다르다.
- 근거 없이 일반화된 문장이다.
- 공개하면 안 되는 내부 맥락을 포함한다.

이런 memory는 바로 삭제하지 않더라도 "확인 필요"로 낮춰야 한다. AI가 답변을 시작할 때 강한 기준처럼 쓰면 안 된다.

## 완료 보고에도 최신 증거를 붙인다

stale context를 막는 마지막 장치는 완료 보고다.

AI가 "완료했습니다"라고 말할 때, 그 근거가 memory인지 최신 증거인지 분리해야 한다. 예를 들어 "이전 기록상 완료"와 "방금 빌드 통과"는 다르다.

좋은 완료 보고는 이렇게 생겼다.

```text
변경한 것: 새 글 3편과 대표 이미지 3장 추가
방금 확인한 것: build 통과, source 민감 패턴 검색 통과
확인하지 않은 것: 원격 배포와 live URL
```

이런 보고는 오래된 기억과 fresh proof를 구분한다. 사용자는 무엇이 지금 확인된 사실인지 볼 수 있다.

## 내가 남긴 기준

AI memory는 반복 설명을 줄여준다. 하지만 오래된 memory를 현재 사실처럼 쓰면 오히려 판단을 망친다.

그래서 memory는 출발점으로만 쓴다. 현재 요청과 충돌하는지 보고, 최신 기록과 실제 파일을 확인하고, 검증 결과를 새로 모은다. 오래된 기준은 필요하면 낮추거나 보류한다.

AI가 잘 기억하는 것만큼 중요한 것은, 기억이 틀릴 수 있다는 전제를 넣는 것이다. stale context를 막는 핵심은 memory를 믿지 않는 것이 아니라, memory를 최신 증거와 함께 쓰는 것이다.

## 확인 범위와 한계

- 이 글은 개인 AI 작업 운영 기준을 공개 가능한 형태로 정리한 것이다.
- 특정 회사, 제품, 고객, 내부 업무 기록, 실제 session state는 포함하지 않았다.
- 글에 포함한 상태 확인 예시는 공개 설명을 위한 합성 예시다.
- 이 글은 memory 시스템 구현 방법이 아니라, 오래된 맥락을 현재 판단에 그대로 쓰지 않기 위한 작업 습관을 설명한다.
