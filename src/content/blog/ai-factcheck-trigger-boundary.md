“확인해줘”라는 단어만 보고 AI의 행동을 정하면 두 방향으로 실패한다. 관찰만 원했는데 파일을 바꿀 수 있고, 문제면 고치라는 요청을 받고도 보고만 한 뒤 멈출 수 있다.

이 글의 질문은 **요청에 들어간 특정 단어가 아니라 목적, 허용된 행동, 실패 비용으로 관찰·가설 검증·실행을 어떻게 구분하는가**다. 같은 문장을 세 문맥에 놓은 합성 사례로 경계가 바뀌는 과정을 살펴본다.

```text
이 설정이 build 지연의 원인인지 확인해줘.
```

문장은 세 번 모두 같다. 하지만 앞서 합의한 목적과 권한이 다르면 완료 조건도 달라진다.

## 먼저 세 축으로 계약을 읽는다

요청을 받으면 다음 세 질문을 순서대로 본다.

1. **목적:** 사용자가 알고 싶은 것은 현재 상태인가, 원인 판단인가, 문제 해소인가.
2. **허용된 행동:** 읽기와 명령 실행만 가능한가, 파일 수정까지 가능한가, 외부 반영도 포함되는가.
3. **실패 비용:** 틀린 보고, 잘못된 수정, 비용 발생, 되돌리기 어려운 변경 중 무엇이 생길 수 있는가.

이 세 축은 작업의 깊이와 중단 지점을 정한다. “확인”이란 단어는 목적의 단서일 수 있지만 수정 권한을 자동으로 만들지는 않는다.

## 문맥 1: 현재 상태를 보고하는 관찰 계약

첫 문맥에서 사용자는 현재 설정값과 최근 build 기록만 알고 싶어 한다. 읽기와 기존 기록 확인은 허용했지만 수정은 요청하지 않았다.

```text
목적: 현재 설정과 관찰된 build 시간을 보고한다.
허용 행동: 설정 읽기, 기존 build 기록 읽기.
허용하지 않은 행동: 파일 수정, 새 비교 실험, 외부 반영.
실패 비용: 원인으로 단정하면 잘못된 후속 결정을 부른다.
```

합성 기록에서 설정은 켜져 있고 최근 build는 `42.0초`였다. 이 문맥의 올바른 답은 여기까지다.

```text
관찰: 설정은 켜져 있고, 확인한 build 기록은 42.0초입니다.
한계: 설정을 끈 비교 결과가 없어 원인인지는 판단할 수 없습니다.
변경: 수행하지 않았습니다.
```

설정과 느린 build가 함께 관찰됐다는 사실만으로 인과관계는 확인되지 않는다. 이 단계에서 설정을 끄면 과잉 실행이고, “원인이 맞다”고 말하면 관찰보다 넓은 결론이다.

## 문맥 2: 원인 가설을 반례와 비교하는 검증 계약

두 번째 문맥에서 목적은 “설정이 지연의 원인인가”를 판단하는 것이다. 영구 수정은 허용하지 않았지만, 같은 합성 project의 임시 복제본에서 설정 하나만 바꾼 비교 실행은 허용했다.

비교 전에 판정 기준을 고정한다.

```text
가설: 설정을 켜면 build 중앙값이 의미 있게 증가한다.
고정 조건: 입력, dependency, 실행 환경, build 명령.
변경 조건: 설정 on/off.
반복: 각 3회.
지지 기준: 중앙값 차이가 2.0초 이상.
중단 조건: 다른 조건이 바뀌거나 실행 결과가 재현되지 않음.
```

합성 실행 결과는 다음과 같다.

| 조건 | 세 번의 build 시간 | 중앙값 |
|---|---|---:|
| 설정 on | 41.9초, 42.0초, 42.1초 | 42.0초 |
| 설정 off | 30.8초, 30.9초, 31.1초 | 30.9초 |

중앙값 차이는 `11.1초`로 미리 정한 `2.0초` 기준보다 크다. 이 합성 조건에서는 “설정이 관찰된 지연의 원인 후보를 지지한다”고 판정할 수 있다. 그래도 이 문맥의 권한은 검증까지다. 실제 파일은 바꾸지 않는다.

비교 결과가 on `31.0초`, off `30.9초`였다면 가설을 지지하지 않는다고 결론내려야 한다. 설정을 원인으로 정해 놓고 작은 차이도 유리하게 해석하면 검증이 아니라 확인 편향이 된다.

## 문맥 3: 확인 뒤 수정하고 재검증하는 실행 계약

세 번째 문맥에서는 같은 요청 전에 다음 권한이 명시돼 있다고 가정한다.

```text
지연 원인이 확인되면 해당 설정 하나를 끄고,
같은 입력과 환경에서 build를 3회 다시 실행한다.

acceptance criteria:
- post-fix 중앙값이 paired baseline보다 2.0초 이상 개선된다.
- 세 번의 build가 모두 성공한다.
- 설정이 관여하는 동작의 focused check가 통과한다.

rollback criteria:
- 위 세 조건 중 하나라도 실패하면 설정을 원래 값으로 되돌린다.
- rollback 뒤 기준 build와 focused check를 다시 실행한다.

외부 반영은 하지 않는다.
```

목적은 원인 보고가 아니라 기능을 보존한 지연 해소다. 허용 행동은 설정 한 항목 수정과 재검증까지고, 중단 조건은 범위 밖 파일 변경, 외부 반영 필요, acceptance 기준 미충족이다. 성능만 좋아지고 설정이 관여하는 동작이 깨지면 성공이 아니다.

앞의 `42.0초`를 같은 round의 paired baseline으로 고정한 합성 실행 결과는 다음과 같다.

```text
post-fix build: 30.8초, 30.9초, 31.1초
post-fix 중앙값: 30.9초
paired baseline 대비 개선: 11.1초
build 성공: 3/3
focused check: pass
  - 유효한 합성 asset이 생성 결과에 포함됨
  - 누락된 합성 asset은 예상대로 거부됨
```

`11.1초` 개선은 사전에 정한 `2.0초` 기준을 넘고, build는 모두 성공했으며, 설정이 관여하는 asset 처리의 focused check도 통과했다. 따라서 이 합성 결과는 세 acceptance criteria를 모두 충족해 수정 상태를 유지한다.

반대로 중앙값 개선이 `2.0초` 미만이거나, 세 build 중 하나라도 실패하거나, focused check가 실패하면 설정을 즉시 원래 값으로 되돌린다. rollback 뒤 기준 build와 같은 focused check가 통과해야 복구를 닫고, 그다음 원인 가설을 다시 검토한다.

이 문맥에서 확인 결과만 보고하고 멈추면 요청을 덜 수행한 것이다. 같은 문장이라도 이미 부여된 수정 권한과 성공·rollback 조건이 행동 계약을 바꾼다.

## 세 문맥의 차이는 trigger가 아니라 계약이다

| 같은 요청의 문맥 | 실제 목적 | 최대 허용 행동 | 완료 조건 |
|---|---|---|---|
| 관찰 | 현재 상태를 안다 | 읽기 | 관찰 범위와 한계 보고 |
| 가설 검증 | 원인 여부를 판단한다 | 통제된 임시 비교 | 지지·반박·증거 부족 판정 |
| 실행 | 기능을 보존하며 확인된 문제를 해소한다 | 명시된 설정 수정과 재검증 | 성능·build·focused check acceptance 또는 rollback |

특정 단어에만 의존하면 이 차이를 잃는다. “확인”이 있으니 항상 읽기 전용이라고 정하면 실행 계약을 놓친다. 반대로 “원인”이나 “문제”가 보인다고 바로 수정하면 관찰 계약을 침범한다.

## 모호할 때는 최소 권한으로 증거를 모은다

문맥을 읽어도 권한이 분명하지 않다면 되돌릴 수 있는 읽기부터 시작한다. 최소 권한은 일을 작게 한다는 뜻이 아니라, 새 권한이 필요한 경계 전까지 의미 있는 증거를 모은다는 뜻이다.

```text
1. 현재 상태와 재현 여부를 읽기 전용으로 확인한다.
2. 원인 판단에 필요한 다음 행동을 적는다.
3. 그 행동이 수정, 외부 write, 비용, 되돌리기 어려운 변화를 포함하는지 본다.
4. 명시되지 않은 경계를 만나면 실행 전에 멈추고 범위를 확인한다.
```

negative trigger도 단어 목록으로 만들 필요가 없다. 다음과 같이 행동의 성질과 연결하면 된다.

- **수정 권한 없음:** 진단 결과를 보고하되 파일을 바꾸지 않는다.
- **외부 write 불명확:** 초안이나 local 검증까지만 하고 전송·발행·배포 전 멈춘다.
- **비용 상한 없음:** 유료 실행이나 대량 작업 전에 예상 범위와 중단 기준을 확인한다.
- **되돌리기 어려움:** 삭제, 데이터 변경, 운영 전환 전에 별도 승인과 복구 조건을 확인한다.

이 기준은 “확인해줘”라는 단어가 있느냐보다 안정적이다. 사용자가 다른 표현을 써도 행동의 권한과 실패 비용을 같은 방식으로 판단할 수 있기 때문이다.

## 반례: “문제면 바로 고쳐줘”인데 보고만 해도 실패다

최소 권한을 “무조건 읽기 전용”으로 오해하면 반대 실패가 생긴다.

```text
이 설정이 build 지연의 원인인지 확인하고, 문제면 바로 고쳐줘.
설정 한 항목 수정과 같은 build 재검증까지 진행해.
```

이 요청은 확인 후 실행 권한을 분명히 준다. 가설이 기준을 충족했고 수정 범위도 한 항목으로 고정됐다면, AI가 “원인이 맞습니다. 수정할까요?”라고 다시 묻고 멈추는 것은 계약 미이행이다. 수정, 재검증, 실패 시 rollback까지 진행해야 한다.

그렇다고 이 문장이 외부 발행이나 다른 설정 변경까지 허용하는 것은 아니다. 명시된 행동을 끝까지 수행하는 것과 권한을 새로 넓히는 것은 구분해야 한다.

## 결론: 요청 경계는 목적·행동·비용의 교집합이다

AI 사실확인 요청의 경계는 “확인해줘” 같은 단어 하나로 정해지지 않는다. 사용자가 얻으려는 결과, 허용한 행동, 잘못했을 때의 비용을 함께 봐야 관찰·가설 검증·실행을 구분할 수 있다.

모호하면 읽기 전용 증거부터 모으고 수정·외부 write·비용·되돌리기 어려운 경계에서 멈춘다. 반대로 사용자가 “문제면 바로 고쳐줘”라고 범위와 검증 조건까지 명시했다면, 확인만 하고 멈추지 말고 그 권한 안에서 실행까지 닫는다. 실행 완료는 성능 개선과 기능 보존이 함께 acceptance 기준을 통과할 때만 선언하고, 하나라도 실패하면 미리 정한 rollback을 수행한다.

허용된 실행도 실패할 수 있다. 권한은 행동해도 되는지를 정하고, 진단은 왜 실패했으며 무엇부터 복구할지를 정한다. 두 판단은 별개이며, 실행이 실패했다면 [AI 작업의 build 실패를 원인과 첫 복구 행동으로 분류하는 법](/blog/blog/ai-work-failure-taxonomy/)으로 first action을 고른다.

## 확인 범위와 한계

- 시간 측정과 설정은 경계 차이를 보여주기 위한 합성 사례이며 실제 성능 결과가 아니다.
- 세 번의 실행과 `2.0초` 기준은 이 사례를 설명하기 위한 값이다. 다른 환경의 성능 판정 기준으로 일반화할 수 없다.
- 법률·보안·운영처럼 별도 승인 체계가 있는 영역에서는 사용자의 한 문장만으로 충분한 권한이 생기지 않을 수 있다.
- 권한이 충돌하거나 누가 승인할 수 있는지 불분명하면 실행하지 말고 권한 소유자와 절차를 확인해야 한다.
