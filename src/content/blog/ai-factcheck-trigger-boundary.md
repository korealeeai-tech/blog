AI에게 "확인해줘"라고 말했는데, 갑자기 수정까지 해버리는 경우가 있다. 반대로 "이 판단이 맞는지 검증해줘"라고 했는데, 파일 하나만 보고 짧게 끝내는 경우도 있다.

둘 다 문제다. 전자는 과잉 실행이고, 후자는 느슨한 확인이다. 사람은 같은 말로 요청했다고 느끼지만, AI에게는 "확인"이라는 단어가 너무 넓게 해석될 수 있다.

그래서 나는 AI에게 사실확인을 맡길 때 요청의 범위를 먼저 나누려고 한다. 단순 확인인지, 가설 검증인지, 확인 후 실행까지 원하는지 구분하면 답변 품질이 훨씬 안정된다.

이 글은 개인 AI 작업에서 쓰는 요청 분류 기준을 정리한 것이다. 특정 도구의 공식 trigger 규칙이 아니라, 사람이 AI에게 일을 맡길 때 생기는 오해를 줄이기 위한 운영 방식이다.

## "확인"에는 서로 다른 일이 섞여 있다

"확인해줘"라는 말은 짧지만 안에는 여러 뜻이 들어갈 수 있다.

파일이 있는지 봐달라는 뜻일 수 있다. 명령 출력이 어떤지 알려달라는 뜻일 수 있다. 내 진단이 맞는지 반례까지 찾아달라는 뜻일 수 있다. 문제가 맞으면 바로 고쳐달라는 뜻일 수도 있다.

사람끼리는 앞뒤 맥락으로 어느 정도 알아듣는다. 하지만 AI에게는 이 구분을 명시해주는 편이 안전하다. 특히 파일 수정, 공개 글 작성, 외부 시스템 변경처럼 되돌리기 어렵거나 공개 표면이 생기는 작업에서는 더 그렇다.

나는 보통 요청을 세 가지로 나눈다.

- 단순 확인
- 가설 검증
- 실행 요청

이 셋은 비슷해 보이지만 성공 기준이 다르다.

## 단순 확인은 관찰까지만 한다

단순 확인은 관찰 가능한 사실을 보는 일이다.

예를 들면 이런 요청이다.

```text
이 파일이 있는지 확인해줘.
현재 branch가 뭔지 봐줘.
빌드가 통과하는지 확인해줘.
이 문구가 남아 있는지 검색해줘.
```

이때 AI는 해석을 크게 넓히지 않는 편이 좋다. 파일이 있는지 보라고 했으면 파일 존재를 확인하고, 필요하면 관련 경로 정도만 덧붙인다. 요청하지 않았는데 구조를 바꾸거나, 원인을 추정해 수정까지 들어가면 과잉 실행이 된다.

단순 확인의 좋은 완료 보고는 짧다.

```text
확인 결과, 해당 파일은 없습니다.
검색 범위는 src/content/blog였고, 유사 파일도 발견하지 못했습니다.
```

확인 범위가 드러나야 한다. 그래야 사용자가 "아, 전체 repo가 아니라 해당 폴더만 본 거구나"라고 판단할 수 있다.

## 가설 검증은 반례까지 본다

가설 검증은 단순 확인보다 넓다. 사용자가 이미 어떤 판단을 갖고 있고, 그 판단이 맞는지 확인하고 싶은 상황이다.

예를 들면 이런 요청이다.

```text
내 생각이 맞는지 검증해줘.
이 원인 분석이 맞는지 반박해봐.
이 접근이 틀렸는지 확인해줘.
```

이때 AI가 단순히 "맞는 것 같습니다"라고 답하면 약하다. 가설 검증에서는 주장을 분해하고, 근거를 찾고, 반례를 확인하고, 부족한 증거를 분리해야 한다.

나는 가설 검증 요청에는 최소한 네 가지가 들어가야 한다고 본다.

- 사용자의 주장을 판단 단위로 나눴는가
- 찬성 근거와 반대 근거를 분리했는가
- 확인한 것과 추정한 것을 구분했는가
- 결론을 내릴 수 없으면 보류했는가

여기서 중요한 것은 보류 가능성이다. 모든 검증이 결론으로 끝나지는 않는다. "현재 근거로는 부족하다"는 답이 가장 정직한 경우도 있다.

## 실행 요청은 확인 뒤 행동까지 포함한다

실행 요청은 확인에서 끝나지 않는다. 확인한 뒤 수정, 반영, 작성, 정리, 배포 같은 행동으로 이어진다.

예를 들면 이런 요청이다.

```text
문제 확인하고 바로 고쳐줘.
검증해서 맞으면 반영해줘.
글 3편을 순차적으로 작성해줘.
```

이 경우 AI는 확인만 하고 멈추면 부족하다. 필요한 범위를 읽고, 변경하고, 검증하고, 마지막에 바뀐 파일과 검증 결과를 보고해야 한다.

다만 실행 요청에도 경계가 필요하다. 특히 외부 발행, 비용 발생, 운영 시스템 변경, 삭제처럼 되돌리기 어려운 작업은 사용자가 명시한 범위 안에서만 해야 한다. "확인해줘"를 "삭제해줘"로 해석하면 안 된다.

실행 요청의 좋은 완료 보고는 결과와 검증을 같이 말한다. 실행하지 않고 멈춘 경계도 함께 남겨야 한다.

```text
변경: 글 3편과 대표 이미지 3장을 추가했습니다.
검증: 빌드 통과, 민감 패턴 검색 통과.
보류: commit/push는 사용자 확인 전에는 하지 않았습니다.
```

이렇게 쓰면 실행한 것과 멈춘 것이 같이 보인다.

## 과발동을 막는 negative trigger

AI 작업에서 trigger를 설계할 때는 "언제 발동할지"만큼 "언제 발동하지 않을지"가 중요하다.

예를 들어 "검증해줘"라는 말이 있으면 항상 깊은 가설 검증을 해야 할까? 그렇지 않다. 사용자가 단순히 명령 결과를 확인해달라는 뜻으로 말했을 수도 있다.

그래서 나는 다음 요청은 가설 검증으로 과하게 해석하지 않는 편이 좋다고 본다.

- 단순 파일 존재 확인
- 명령 실행 결과 확인
- 일반적인 코드 리뷰
- 순수 요약
- 예측만 원하는 질문
- 구현을 우선하라고 명시한 요청

반대로 사용자가 "내 생각", "내 가설", "이 접근이 틀렸는지", "반례", "팩트체크"처럼 자신의 판단을 검증해달라고 말하면 가설 검증으로 보는 것이 맞다.

이 구분이 있어야 AI가 매번 장황해지지 않는다. 작은 확인에는 작게 답하고, 판단 검증에는 깊게 들어가고, 실행 요청에는 끝까지 처리할 수 있다.

## 요청 문장을 조금만 바꾸면 달라진다

사용자 입장에서도 질문을 조금만 다르게 쓰면 결과가 좋아진다.

단순 확인을 원하면 이렇게 쓴다.

```text
수정하지 말고, 현재 상태만 확인해줘.
확인 범위와 결과만 알려줘.
```

가설 검증을 원하면 이렇게 쓴다.

```text
내 판단이 맞는지 검증해줘.
찬성 근거, 반례, 증거 부족 항목을 나눠줘.
```

실행까지 원하면 이렇게 쓴다.

```text
확인한 뒤 문제가 맞으면 바로 수정하고, 검증 결과까지 보고해줘.
```

이 정도만 명시해도 AI의 행동 범위가 훨씬 선명해진다.

## 내가 남긴 기준

AI에게 사실확인을 맡길 때는 "확인"이라는 단어 하나에 기대지 않는다. 단순 확인, 가설 검증, 실행 요청을 나눠서 본다.

단순 확인은 관찰까지만 한다. 가설 검증은 반례와 보류 조건까지 본다. 실행 요청은 변경과 검증까지 이어가되, 사용자가 명시하지 않은 위험한 행동은 멈춘다.

좋은 AI 운영은 더 많은 일을 자동으로 시키는 것이 아니다. 지금 요청이 어디까지의 일인지 서로 오해하지 않게 만드는 것이다.

## 확인 범위와 한계

- 이 글은 개인 AI 작업 운영 기준을 공개 가능한 형태로 정리한 것이다.
- 특정 도구의 내부 라우팅, trigger 구현, 시스템 프롬프트를 설명하지 않는다.
- 글에 들어간 요청 문장은 설명을 위한 일반 예시이며, 실제 private 작업 지시를 옮긴 것이 아니다.
- 의료, 법률, 금융, 보안처럼 실패 비용이 큰 주제는 단순 AI 검증으로 결론내리지 말고 최신 1차 출처와 전문가 검토를 함께 봐야 한다.
